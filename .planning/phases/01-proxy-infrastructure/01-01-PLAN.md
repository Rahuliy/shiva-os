<objective>
Create and deploy a Cloudflare Worker that proxies Teller API calls (sandbox mode, plain fetch) and stubs eBay API routes, with CORS headers, one-retry error handling, and structured error responses. Wire both lakshmi.html and kubera.html to read the Worker URL from localStorage.

Output: Deployable Worker in `worker/`, browser-side Worker URL config in both HTML pages, .gitignore protecting secrets.
</objective>

**worker/src/index.js** — Single Worker entry point implementing:

1. **`corsHeaders(request)`** — Returns CORS headers for every response. `Access-Control-Allow-Origin` mirrors the request's `Origin` header (or `*` if absent). Allows methods: `GET, POST, OPTIONS`. Allows headers: `Content-Type, Authorization`. Sets `Max-Age: 86400` and `Vary: Origin`. Also sets `Content-Type: application/json`.

2. **`handleOptions(request)`** — CORS preflight handler. If request has all three preflight indicators (Origin, Access-Control-Request-Method, Access-Control-Request-Headers), return 204 with corsHeaders. Otherwise return plain 200 with `Allow: GET, POST, OPTIONS`.

3. **`fetchWithRetry(upstreamRequest)`** — Attempt fetch once. If network error or non-ok status, retry once with `upstreamRequest.clone()`. If both fail, return `null`.

4. **`errorResponse(request, message, code)`** — Returns a `Response` with JSON body `{error: message, code: code}`, status set to `code`, and corsHeaders.

5. **`handleTeller(request, env, url)`** — Teller sandbox proxy:
   - Strip `/api/teller` prefix from pathname to get the Teller API path
   - Build upstream URL: `https://api.teller.io${tellerPath}${url.search}`
   - Create upstream Request with method from original request, forwarding `Authorization` and `Content-Type` headers (do NOT forward `Origin` header — Teller may reject non-standard origins)
   - Call `fetchWithRetry`; if null, return `errorResponse(request, 'Teller upstream unavailable', 502)`
   - Read upstream response body as text, return new Response with upstream status and corsHeaders
   - Add a comment block documenting the production mTLS switch: `env.TELLER_CERT.fetch(tellerUrl, options)` instead of plain `fetch`

6. **`handleEbay(request, env, url)`** — eBay stub (approval pending):
   - Return structured 503: `{error: 'eBay integration pending approval', code: 503}` with corsHeaders
   - Add a comment block with the full eBay OAuth client credentials grant implementation (from RESEARCH.md Pattern 5) so the production code is ready to uncomment when approval arrives. The stub checks `env.EBAY_APP_ID` — if the secret exists, it runs the real flow; if not, it returns the 503 stub.

7. **`export default { fetch(request, env, ctx) }`** — Main entry:
   - OPTIONS → `handleOptions(request)`
   - Parse URL. `/api/teller/*` → `handleTeller`. `/api/ebay/*` → `handleEbay`.
   - Anything else → `errorResponse(request, 'Not found', 404)`

Use module Worker syntax (`export default`), NOT Service Worker syntax (`addEventListener`).
Use `wrangler deploy` (NOT deprecated `wrangler publish`).

In both `lakshmi.html` and `kubera.html`, add a Worker URL configuration block near the top of the inline `<script>` section (before any existing JavaScript logic). The pattern:

```javascript
// ── Proxy Configuration ──────────────────────────────────
const WORKER_URL = localStorage.getItem('shiva-worker-url') || '';

function workerFetch(path, options = {}) {
  if (!WORKER_URL) {
    console.warn('Worker URL not configured. Set localStorage key "shiva-worker-url" to your Cloudflare Worker URL.');
    return Promise.resolve(new Response(JSON.stringify({ error: 'Worker URL not configured', code: 0 }), { status: 0 }));
  }
  return fetch(`${WORKER_URL}${path}`, options);
}
```

This provides:
1. **`WORKER_URL`** constant read from localStorage with empty string fallback (no hardcoded default URL — user MUST configure it)
2. **`workerFetch(path, options)`** helper that prepends the Worker URL to any path and returns the fetch promise. If WORKER_URL is not set, it logs a console warning and returns a synthetic error response instead of making a failing network call.

Do NOT add any actual API calls yet (those come in Phase 3+). Only add the configuration plumbing.

Place the block immediately inside the first `<script>` tag in each file, before any existing code. Add a single blank line after the block to separate it from existing code.

Do NOT modify any existing JavaScript logic, styles, or HTML structure in either file. Only ADD the new block.
